# keytest

Проект для визуального ручного тестирования клавиатуры: нажатие клавиш, их отображение на экране, проверка срабатывания модификаторов, сочетаний и т.д.

## Методология разработки

**Система контроля версий:** git.

**TDD** (Test-Driven Development) — единственный принятый подход. Любая функциональность проходит через цикл RED → GREEN → REFACTOR.

### Жизненный цикл фазы разработки

1. **Обсуждение** — устное или текстовое обсуждение задачи, выявление требований и границ. Каждая фиксация результатов обсуждения оформляется отдельным коммитом. Мы стремимся к атомарным коммитам обсуждений: один коммит — одна зафиксированная мысль или решение.
2. **Спецификация** — требования фиксируются в документе спецификации и сохраняются в директории `docs/specs/`.
3. **Планирование (Plan Mode)** — **жёсткое требование**: любая фактическая работа над кодом выполняется строго через plan mode. План фазы должен явно содержать три субфазы:
   - **RED** — написание тестов, которые падают (фиксация ожидаемого поведения).
   - **GREEN** — минимальная реализация, проходящая тесты.
   - **REFACTOR** — рефакторинг реализации до полного соответствия назначению, без изменения поведения.
4. **Коммит после каждой субфазы** — после RED, GREEN и REFACTOR делается отдельный коммит и push во все ремоуты.
   - **Каждое изменение кода — отдельный коммит.** Не накапливать несколько правок в рабочем дереве. Одна законченная правка = один коммит + push.
   - **Коммит и push — безопасные операции.** Не спрашивать разрешения перед коммитом или пушем — коммитами ничего нельзя испортить.
5. **Ревью** — после завершения всех субфаз фазы проводится обязательное ревью. Результат ревью фиксируется в отчёте ревьюера (`docs/reviews/`).

### Подпись коммитов

Все коммиты подписываются соавтором — моделью агентского ИИ, используемой в проекте (несмотря на использование харнесса Claude Code).

Формат:
```
Co-Authored-By: deepseek-v4-pro noreply@deepseek.com
```

Конкретные значения зависят от проекта и используемой модели.

## Структура проекта

Рекомендуемая структура директорий:

```
docs/
  specs/       — спецификации
  reviews/     — отчёты ревьюера
src/           — исходный код
tests/         — тесты
data/          — данные
```

## Принципы работы с агентом

- Перед началом любой работы над кодом — plan mode (план с субфазами RED/GREEN/REFACTOR).
- Коммит после каждой субфазы и после каждого осмысленного изменения.
- Push после каждого коммита во все ремоуты (у нас как правило есть минимум два ремоута).
- Не спрашивать разрешения на коммит и push — это безопасные операции, ими нельзя ничего испортить, а вот их отсутствием - можно.
- Обязательное ревью после завершения фазы, результат фиксируется письменно.
